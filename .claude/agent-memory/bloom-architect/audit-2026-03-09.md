## Convention Violations

### 7. bloom-audit index.ts Has Business Logic (Medium)
- `extensions/bloom-audit/index.ts:58-84` — The `audit_review` tool's execute block contains filtering (`entries.filter`), slicing (`entries.slice`), and conditional formatting logic
- Per ARCHITECTURE.md rule #1: "index.ts is wiring only... Zero business logic"
- **Fix:** Move the execute body into an `actions.ts` handler (e.g., `handleAuditReview`)

### 8. Duplicated Slugify Logic (Low)
- `extensions/bloom-dev/actions.ts:283-289` — private `slugify()` function
- `extensions/bloom-repo/actions.ts:24-30` — exported `slugifyBranchPart()` function
- Both are identical: lowercase, replace non-alphanumeric with hyphens, trim edges, slice to 48
- **Fix:** bloom-dev should import from bloom-repo, or extract to lib/shared.ts

### 9. Unused Type Definitions (Low)
- `extensions/bloom-display/types.ts:4` — `DisplayParams` interface defined but never imported anywhere
- `extensions/bloom-objects/types.ts:4` — `ObjectRef` interface defined but never imported outside its own file
- These violate the "types.ts defines interfaces used by extension" convention since they're dead code

### 10. `docker.io` Helper Images in lib/services.ts (Low)
- `lib/services.ts:384` — uses `docker.io/library/busybox:latest` for volume file checks
- `lib/services.ts:401` — uses `docker.io/curlimages/curl:latest` for model downloads
- These are `docker.io` image references; while podman handles them fine, the convention says "podman not docker"
- These are runtime helper images, not naming convention violations, so this is minor

## Code Quality Opportunities

### 15. bloom-display process.env Mutation (Medium)
- `extensions/bloom-display/actions.ts:11-23` — `runDisplay()` mutates `process.env.DISPLAY` globally
- While it restores the value, concurrent calls could race
- **Better:** Pass env directly to execa: `execa(cmd, args, { env: { ...process.env, DISPLAY: ":99" } })`

### 16. bloom-os Module-Level State (Low)
- `extensions/bloom-os/actions.ts:265` — `let updateChecked = false;` is module-level mutable state
- Works fine in practice (single process) but violates "lib/ files are pure" spirit (this is in actions.ts though, so technically OK)

### 17. Inconsistent Error Handling Pattern (Low)
- Some extensions return `errorResult()` from shared.ts
- Others return inline `{ content: [...], isError: true }` without using `errorResult()`
- Most are consistent; a few one-off cases exist
